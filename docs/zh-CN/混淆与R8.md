# 混淆与 R8

面向：**在应用模块开启 `minifyEnabled true`（R8 / ProGuard）** 的集成方。

---

## 1. SDK 已声明的 consumer 规则

`smart` 与 `node` 的 `build.gradle` 中均配置了：

```groovy
consumerProguardFiles "consumer-rules.pro"
```

当前两个模块下的 **`consumer-rules.pro` 文件为空**，表示：

- 库作者 **未** 在 AAR 内附带额外 `-keep` 规则；  
- **不表示**「你的 App 开混淆一定零配置」——若你在业务里对 **本库涉及的类** 做了反射、序列化按字段名匹配等，仍需按 Android 通用规范自行补充规则。

---

## 2. 一般集成（仅调用公开 API、无反射）

多数情况下：

1. 使用本库提供的 **Adapter / Provider / ViewBinding** 正常调用；  
2. **不要** 对 `pw.xiaohaozi.xadapter` 整包做无脑 `-keep`（会增大包体）；  
3. 若 Release 出现 **ClassNotFoundException / NoSuchMethodError**，用 **R8 映射文件（mapping.txt）** 对照堆栈，再考虑最小化 `-keep`。

若混淆后仅有 **资源或 ViewBinding 生成类** 相关崩溃，先检查是否对 **ViewBinding 绑定类** 做了反射；若有，需对对应 `Binding` 类按需 keep（建议缩小到具体类名）。

---

## 3. 若你使用 Gson / Moshi 等与数据类强耦合

本库 **不负责** 为你的 JSON 模型写混淆规则。  
若列表数据类同时作为 Gson 等反序列化目标，请按你所用序列化库的 **官方 R8 说明** 配置（例如 `@SerializedName`、Proguard consumer 等），与是否使用 XAdapter **无关**。

---

## 4. 调试建议

1. 先在 **debug（不开混淆）** 下确认功能正常。  
2. 再开 **release + minify**，用 **mapping** 还原堆栈定位。  
3. 仅在确认某类被错误裁剪后，再添加 **最小范围** 的 `-keep`。

---

## 5. 需要向库作者反馈的情况

若你 **未使用反射**、仅按文档调用，在 **最新 Tag** 下仍出现「仅混淆后出现、且堆栈指向库内部公开类被删改」的情况，请到 [Issues](https://github.com/xiaohaozi9825/XAdapter/issues) 附上：

- AGP / Gradle / Kotlin 版本  
- `minifyEnabled`、是否全量 R8  
- 脱敏后的 `mapping` 相关片段与堆栈  

便于评估是否应在后续版本的 `consumer-rules.pro` 中补充规则。

# 混淆与 R8

面向：**在应用模块开启 `minifyEnabled true`（R8 / ProGuard）** 的集成方。

---

## 1. 为什么需要额外配置

XAdapter 在运行时会通过 **泛型签名（Signature）** 和 **ViewBinding 反射** 完成布局创建与多类型匹配，例如：

- `XProvider` 读取子类泛型中的 `ViewBinding` 类型，再反射调用 `inflate(...)`；
- `XAdapter.getItemViewType` 将列表数据的 `Class` 与 Provider 泛型中的 `D` 类型比对；
- `EventImpl` / `AdapterSelectedImpl` 读取 Provider 泛型，跳过头尾/空布局或限定可选类型。

若 Release 混淆后未保留 **Signature** 或 **ViewBinding.inflate**，常见表现为：

- `必须明确指定VB泛型类型`
- `必须明确指定 D 泛型类型`
- `NoSuchMethodException: inflate`
- 多类型列表 item 布局错乱或始终命中第一个 Provider

---

## 2. SDK 已附带的 consumer 规则

`smart` 与 `node` 的 `build.gradle` 中均配置了：

```groovy
consumerProguardFiles "consumer-rules.pro"
```

依赖 AAR 后，以下规则会 **自动合并** 到集成方的 Release 构建中（无需手动复制）：

### smart 模块（`smart/consumer-rules.pro`）

- 保留 `Signature`、`InnerClasses`、`EnclosingMethod`
- 保留 `pw.xiaohaozi.xadapter.smart.**` 包（含 `HEADER` / `FOOTER` / `EMPTY` / `DEFAULT_PAGE` 等占位实体）
- 保留业务代码中 `extends XProvider` / `SmartProvider` / `SmartAdapter` / `XAdapter` 的子类（含 `createAdapter`、`withType`、`addHeader` 等 inline 生成的匿名类）
- 保留所有 `ViewBinding` 实现类的 `inflate` / `bind` 静态方法

### node 模块（`node/consumer-rules.pro`）

- 保留 `pw.xiaohaozi.xadapter.node.**` 包
- 保留业务代码中 `extends NodeProvider` / `NodeAdapter` 的子类（含 `nodeAdapter`、`withType` 等 inline 生成的匿名类）

> **说明**：`node` 依赖 `smart`，引入 `node` 时会同时合并 `smart` 的 consumer 规则。

## 3. 集成方仍需补充的配置

SDK 无法预知你工程的 **DataBinding / ViewBinding 生成类包名**，请在 **应用模块** 的 `proguard-rules.pro`（或等效配置）中，将下面占位包名替换为你自己的：

```proguard
# 将 com.yourcompany.yourapp 替换为你的 applicationId / 实际生成包名
-keep class com.yourcompany.yourapp.databinding.** { *; }
```

如何确认包名：在工程中搜索任意一个 `*Binding` 类（如 `ItemUserBinding`），查看其 `package` 声明即可。

> **注意**：不要使用其他项目的包名（例如 Demo 中的 `pw.xiaohaozi.callnumber.databinding`），必须写 **你自己 App** 的包路径。

若已使用 SDK 附带的 `ViewBinding` 通用保留规则，且 AGP 默认 R8 配置未裁剪 `inflate`，部分工程可以 **不写** 上述包名规则也能正常运行；若 Release 仍出现 `inflate` 相关崩溃，再补上这一条即可。

## 4. 推荐完整配置（可复制）

以下为「SDK consumer 规则 + 业务 ViewBinding 包名」的完整示例，便于对照或在不使用 AAR consumer 合并的旧版本上手动添加。

### 仅使用 smart

```proguard
# ---------- 由 SDK consumer-rules.pro 自动合并，一般无需手写 ----------
-keepattributes Signature, InnerClasses, EnclosingMethod

-keep class pw.xiaohaozi.xadapter.smart.** { *; }

-keep,allowobfuscation class * extends pw.xiaohaozi.xadapter.smart.provider.XProvider { *; }
-keep,allowobfuscation class * extends pw.xiaohaozi.xadapter.smart.provider.SmartProvider { *; }
-keep,allowobfuscation class * extends pw.xiaohaozi.xadapter.smart.adapter.SmartAdapter { *; }
-keep,allowobfuscation class * extends pw.xiaohaozi.xadapter.smart.adapter.XAdapter { *; }

-keepclassmembers class * implements androidx.viewbinding.ViewBinding {
    public static ** inflate(android.view.LayoutInflater, android.view.ViewGroup, boolean);
    public static ** inflate(android.view.LayoutInflater);
    public static ** bind(android.view.View);
}

# ---------- 集成方必改：自己的 ViewBinding 生成包 ----------
-keep class com.yourcompany.yourapp.databinding.** { *; }
```

### 同时使用 smart + node

在上一节基础上，SDK 还会自动合并：

```proguard
-keep class pw.xiaohaozi.xadapter.node.** { *; }

-keep,allowobfuscation class * extends pw.xiaohaozi.xadapter.node.NodeProvider { *; }
-keep,allowobfuscation class * extends pw.xiaohaozi.xadapter.node.NodeAdapter { *; }
```

### 在 app 模块中启用混淆

```groovy
android {
    buildTypes {
        release {
            minifyEnabled true
            shrinkResources true   // 可选
            proguardFiles getDefaultProguardFile('proguard-android-optimize.txt'), 'proguard-rules.pro'
        }
    }
}
```

## 5. 通常不需要额外 keep 的内容

| 类型 | 说明 |
|------|------|
| 列表数据类（`User`、`Order` 等） | 走 `equals` / DiffUtil，框架不按字段名反射；多类型匹配依赖 Provider 泛型 Signature，不是数据类类名 |
| `XMultiItemEntity` | 通过 `getItemViewType()` 返回类型，不依赖类名字节码 |
| 对整个 `pw.xiaohaozi.xadapter` 无脑 `-keep` | SDK consumer 已覆盖 `smart` / `node`；勿重复扩大范围 |

---

## 6. Gson / Moshi 等 JSON 序列化

本库 **不负责** 为你的 JSON 模型编写混淆规则。

若列表数据类同时作为 Gson、Moshi 等反序列化目标，请按你所用序列化库的 **官方 R8 说明** 单独配置（例如 `@SerializedName`、adapter 的 consumer ProGuard 等），与是否使用 XAdapter **无关**。

---

## 7. 特殊 API 说明

### `singleSelect(..., permittedTypes = arrayOf(SomeData::class.java))`

传入的 `Class` 对象用于与 Provider 泛型参数比对。常规数据类无需额外 keep；若你对数据类做了极端裁剪导致 `Class` 不可用，再按需保留对应类。

### 可选：`kotlin-reflect`

`XAdapter` 在检测「可空泛型」时会尝试使用 Kotlin 反射；未引入 `kotlin-reflect` 时会自动降级，**不影响** 混淆配置，也 **不需要** 为其单独添加 keep 规则。

---

## 8. 调试建议

1. 先在 **Debug（不开混淆）** 下确认列表、多布局、头尾布局、空布局等功能正常。
2. 再开 **Release + minify**，用 **mapping.txt** 还原堆栈定位。
3. 仅在确认某类被错误裁剪后，再添加 **最小范围** 的 `-keep`。
4. 若仅 ViewBinding 相关崩溃，优先检查第 3 节的 **databinding 包名** 是否写对。

---

## 9. 需要向库作者反馈的情况

若你已使用 **最新 Tag**、按本文配置 ViewBinding 包名，仍出现「仅混淆后出现、且堆栈指向库内部公开类被删改」的情况，请到 [Issues](https://github.com/xiaohaozi9825/XAdapter/issues) 附上：

- AGP / Gradle / Kotlin 版本
- `minifyEnabled`、是否全量 R8
- 脱敏后的 `mapping` 相关片段与堆栈
- 使用的是 `smart` 还是 `smart + node`，以及 `createAdapter` / `withType` / 自定义 Provider 等用法简述

便于评估是否应在后续版本的 `consumer-rules.pro` 中补充规则。
